## Testing

- Tests are located in `tests/` directory
- Run all tests: `uv run pytest`
- Run specific tests: `uv run pytest tests/test_example.py -v`
  - Add `-x` to exit on first failure
  - Add `-v` for verbose output
  - Add `-vv` for very verbose output

## Code Style

This project strictly follows the [Google Python Style Guide](https://google.github.io/styleguide/pyguide.html). Key rules:

- **Line length**: 100 characters
- **Python version**: 3.10+
- **Linting**: Ruff with extensive rule set (configured in `pyproject.toml`)
- **Formatting**: Ruff formatter (Black-compatible)
- **Type checking**: mypy with strict settings (configured in `pyproject.toml`)
- **Import sorting**: Handled by Ruff
- **Type hints**: Required on all function signatures
- **Docstrings**: Google style format

```python
def example_function(param1: str, param2: int) -> dict:
    """Short description of the function.

    Args:
        param1: Description of param1.
        param2: Description of param2.

    Returns:
        Description of the return value.

    Raises:
        ValueError: When param2 is negative.
    """
    pass
```

### Linter rule suppression (`# noqa`) policy

- **Never** add `# noqa` to silence a linter rule. Fix the underlying code instead.
- Catch specific exception types (`ValueError`, `KeyError`, …) — never bare `except Exception`.
- If a rule genuinely cannot be satisfied, discuss it with the team before suppressing.

## Security

This project includes several security checks run as part of the pre-commit hooks and CI/CD pipeline:

- **Dependency Scanning:** Dependencies are scanned for vulnerabilities using `uv-secure`.
- **License Compliance:** Python package licenses are checked against the project's license strategy.
- **Static Application Security Testing (SAST):** Python code is scanned using the S ruleset (flake8-bandit) in `ruff`.

## Available Skills

Skills are reusable guided workflows located in the `skills/` directory. Each skill has a `SKILL.md` defining when and how to use it, and may include reference files in a `references/` subdirectory.
If not already done, you should copy the `skills/` directory to the `./.<ai-tool-name>/skills/` directory within the AI tool the user is using. Example: `.cursor/skills/`, `.claude/skills/`, `.codex/skills/`, `.github/skills/`.

| Skill | Location | Trigger | Description |
|-------|----------|---------|-------------|
| **update-docs** | `skills/update-docs/` | User asks to update docs, check docs for a PR, sync docs with code, scaffold or review documentation completeness, or mentions README, CHANGELOG, CONTRIBUTING, API reference, or `.md` files. | Guided workflow for updating project documentation based on code changes. Analyzes the active branch, classifies changes by type, and suggests updates to READMEs, CHANGELOGs, API specs, configuration guides, and more. References `CHANGETYPE-SUGGESTIONS.md` for per-type documentation templates. |
| **git-workflow** | `skills/git-workflow/` | User works with git branches, prepares commits, or reviews git history. | Git workflow conventions including branch naming, pre-commit checks, commit message format (conventional commits), and amending commits. |
| **refactoring-code** | `skills/refactoring-code/` | User asks to refactor code, improve readability, performance, maintainability, reduce technical debt, eliminate code smells, or modernize legacy Python code. | Systematic Python refactoring workflow: one change at a time, tests first, references for common refactorings, modern Python patterns, and performance patterns. |

When a user request matches a skill trigger, follow the full workflow defined in that skill's `SKILL.md` instead of ad-hoc responses. Reference the skill's supporting files (e.g., `references/CHANGETYPE-SUGGESTIONS.md`) for structured guidance.
